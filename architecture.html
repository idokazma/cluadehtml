How the system works

A wrapper around Claude Code — call it stack — that turns every agent session into a live HTML page of interactive components. Two agents are involved: the main agent does the work the user asked for; a smaller interface agent watches its event stream and decides what HTML component best expresses each chunk. The rendering is not a static lookup table — it is a model call with a component vocabulary. And the page is bidirectional: clicks, picks, slider drags, and edits become inputs that flow back into the main agent's session.

This page draws all of that out: the boxes, the lines between them, what flows along each line, and how an HTML component is born — and acted on.

1The big picture

Two agents, two flow directions. The main agent is the Claude Code session that does the work — it streams events (tool_use, tool_result, text). The interface agent sits beside it and decides, for each chunk, what HTML component best expresses it. Components stack into a live page; user interactions on those components flow back the other way as synthesized session inputs.

Architecture · two agents, bidirectional flow
human you types prompts · clicks components · drags sliders · scrolls history

Two ingest modes share the same downstream. stack run spawns claude -p --output-format stream-json and parses its stdout. stack attach doesn't spawn anything — it tails the JSONL Claude Code writes for any interactive session, so you can run claude normally in a terminal and the browser is just an enriched view. Below that line, everything is identical.

2Following one event end to end

Click a step to see the data at that point. This is what happens when the model decides to read a file.

Event lifecycle · model emits tool_use(Read)
step 1 model emits tool_use(Read)
step 2 claude stdout stream-json line
step 3 wrapper parses decode JSON event
step 4 mapper resolves → file-viewer
step 5 emit mutation append to log + SSE
step 6 browser appends new component
step 7 tool_result patch the same id

Two key invariants. Stable ids. Every mutation carries an id derived from the model's tool_use_id, so the tool_result that arrives later patches the same component — no flicker, no re-mount. Idempotent log. The browser state is exactly the fold over the mutation log. Reload the page and we replay the log; nothing else is needed.

3The interface agent is an editor

Not every event the main agent fires deserves a component. An agent that reads 20 files to fix one line of code should surface that line and its fix — not 20 file viewers. The interface agent's first question for every event isn't "what component is this?" but "does this earn a place on the page?" Most internal tool calls fold into a single live activity indicator; only events that earn the surface graduate into committed components.

Sessions are narrations, not logs. A log is exhaustive: every tool call rendered, easy to skim and easy to drown in. A narration is editorial: the user sees the structure of the work, the decisions, and the outputs — with the means available to drill deeper when they care.

Three layers (revised)

The same three layers as before, but with the first reframed as a buffer, not a renderer. The interface agent decides what gets committed; the prefilter mostly just stages.

layerroletypical action
1 · prefilter buffer + activity updates the indicator; auto-commits only for: TodoWrite (plan), explicit long-running Bash, things the user has opted to follow
2 · interface agent editorial pass runs each settled chunk; outputs an activity update or 0+ committed components; can replace / enrich / insert
3 · explicit MCP self-authored UI main agent calls render_* when it knows the structure (rare; bypasses editorial pass)

What it looks like in practice

// incoming: a Read call exploring a file { kind: "tool_use", tool: "Read", input: { file_path: "src/charge.ts" } } { kind: "tool_result", text: "…" } // interface agent's editorial pass: { activity: { name: "Looking at charge.ts", metric: "3 of 14" } } // ↑ no committed component. just updates the indicator. // incoming: the diff that fixes the bug { kind: "tool_use", tool: "Edit", input: { old, new } } { kind: "tool_result", text: "ok" } // interface agent's editorial pass: [ { type: "diff", props: { filename, hunks } } ] // ↑ deliverable. commits it. activity indicator clears. // incoming: prose offering 3 options { kind: "text", text: "There are 3 reasonable ways..." } // interface agent's editorial pass: [ { type: "decision", props: { question, options[] } } ] // ↑ decision point. commits a decision card.

4The page is bidirectional · components are inputs

The whole point of replacing the markdown terminal is so the user can act on what they see. Today, when Claude Code asks "do you want A, B, or C?" you type back which one. With components, you click. The click becomes a structured action that the server synthesizes into a follow-up user message and routes back into the main agent's session.

From action to session input

The server's POST /api/action handler synthesizes a follow-up user message. Two strategies, picked by action kind:

// strategy 1 · reference-and-narrate (decisions, picks, toggles) "[interaction with @dec-7] Picked option 2 of 3: Manual freeze — user taps 'freeze today' before midnight." // strategy 2 · verbatim insertion (free-form prompts, edits) "[reference: @code-5 (src/components/HabitCard.tsx)] Use this version instead: <the user's typed text or edited code>"

The synthesized message refers to the component by @id, so the main agent has full context — it produced that component earlier in the session. The message enters the loop:

  • stack run mode — server calls claude --resume <session> -p "<synthesized>" to start the next turn.
  • stack attach mode — server prints the message into the user's terminal (or pushes via Channels) for the live session to pick up.
  • Agent SDK mode (v2) — the wrapper owns the loop; the synthesized message just gets appended.
Round-trip · user picks an option on a decision component
1 user clicks opt 2 on @dec-7
2 optimistic card marks "picked" immediately
3 browser POST /api/action
4 server synthesizes user msg referring to @dec-7
5 queued / sent waits for current turn; then --resume -p
6 new components flow through prefilter + interface agent

Two subtleties: old components stay live across turns — nothing about them is read-only, because the conversation can always re-touch them by id. And actions queue by default — clicking during an in-progress turn doesn't interrupt; the action is held until the turn settles and becomes the next user message.

5How HTML actually gets produced

There is no template engine and no React. The page state is exactly the fold over a mutation log. Each mutation either appends a component, patches one by id, or streams more content into one. The browser keeps a Map<id, { type, props, dom }> and applies mutations as they arrive.

The four mutation ops

// 1. Append a new component to the stack
{ op: "append", parent: "stream", component: { id: "plan-3", type: "plan", props: {…} } }

// 2. Patch a component's props (used when tool_result arrives)
{ op: "patch",  id: "plan-3", props: { items: [...] } }

// 3. Stream content into a component (Bash output, etc.)
{ op: "stream", id: "term-7", append: "line of output\n" }

// 4. Mark a component as done (terminal exit code, test pass/fail count)
{ op: "finalize", id: "term-7", meta: { exit: 0 } }

How a component is born — the producing process

  1. An agent event comes in (tool_use, tool_result, or text block).
  2. The mapper looks up the handler for that event (built-in table or render_* schema).
  3. Handler returns one or more typed mutations (append, patch, stream).
  4. Each mutation is written to .stack/sessions/<id>.events.jsonl and pushed to all browser tabs subscribed via SSE.
  5. In the browser, the component renderer for that type reads props and produces DOM. Same renderer handles initial mount and subsequent patches — the demo's refreshPlan(id) pattern, generalized.
  6. On a fresh page load, the server's /events endpoint replays the entire log before going live. The page reaches the same final state no matter when you join.
Why no framework. Components are independent of each other, each owns its own mutation logic, and the renderer-per-type pattern fits ~30 lines of vanilla JS per component. React would add bundle weight and a diffing layer for something that is already a single-direction diff stream.

6From sessions to the project page

Each session writes a per-session event log. The project page is the aggregate view across all sessions in .stack/sessions/, plus live git/CI state. It is itself a stack of components, generated fresh on each load (it's cheap — these aren't live-mutating).

Aggregation · session logs → project page
inputs session logs .stack/sessions/*.events.jsonl
inputs git state branches · uncommitted · diff vs main
inputs gh / CI PRs · workflow runs · review comments
build aggregator scans, summarizes, ranks
render project.html component stack

What the aggregator scans for in each session log:

  • Decisions resolved or open — every decision component with its picked answer.
  • Files touched — collected from diff components.
  • Test status — last tests finalize event.
  • Outcome — was the session abandoned, completed, or did it ship?
  • Title + summary — the agent's last note, or a one-line synthesis.
