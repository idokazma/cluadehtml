How the system works

A wrapper around Claude Code — call it stack — that turns every agent session into a live HTML page of interactive components. Each tool call, text reply, and tool result becomes (or updates) a component in a single stacked document. This page draws all of that out: the boxes, the lines between them, what flows along each line, and how an HTML component is born from an agent event.

1The big picture

Six moving parts. The user prompts; Claude Code emits events; a small wrapper ingests those events and decides what to render; the mapper turns each event into a typed mutation; the mutations are appended to a per-session event log and pushed live to the browser over SSE. Hover any box to highlight its connections.

Architecture · all flows
human you types prompts · clicks components · scrolls history

local server · node stack server SSE /events · POST /api/prompt · POST /api/action

Two ingest modes share the same downstream. stack run spawns claude -p --output-format stream-json and parses its stdout. stack attach doesn't spawn anything — it tails the JSONL Claude Code writes for any interactive session, so you can run claude normally in a terminal and the browser is just an enriched view. Below that line, everything is identical.

2Following one event end to end

Click a step to see the data at that point. This is what happens when the model decides to read a file.

Event lifecycle · model emits tool_use(Read)
step 1 model emits tool_use(Read)
step 2 claude stdout stream-json line
step 3 wrapper parses decode JSON event
step 4 mapper resolves → file-viewer
step 5 emit mutation append to log + SSE
step 6 browser appends new component
step 7 tool_result patch the same id

Two key invariants. Stable ids. Every mutation carries an id derived from the model's tool_use_id, so the tool_result that arrives later patches the same component — no flicker, no re-mount. Idempotent log. The browser state is exactly the fold over the mutation log. Reload the page and we replay the log; nothing else is needed.

The mapper has two layers. The first is a fixed table for built-in Claude Code tools. The second is the render_* MCP server, which lets the model ask for bespoke components by calling tools.

Layer 2 · the render_* MCP server

The interesting components in the mockup — decision cards, comparison tables, design explorations, theme playgrounds, schema diagrams — aren't standard tool calls. So we ship an MCP server, stack-mcp, that exposes render_* tools. When the model wants one of these, it calls the matching tool; the wrapper sees the tool_use, decodes the payload against a known schema, and renders the component. The model effectively has "speak in components" as part of its toolbox.

4User actions go back the other way

Components are interactive. When the user picks a decision option, drags a slider, clicks "copy as prompt", or right-clicks "ask about this", the page POSTs to the server. The server either synthesizes a follow-up user message (and re-runs claude with --resume <id> -p "<text>") or, when running in attach mode, writes a message into the live session.

Round-trip · user picks an option on a decision component
1 user click opt 2 on @dec-7
2 browser POST /api/action
3 server synthesizes "user picked option 2 (manual freeze) for @dec-7"
4 re-run claude --resume <id> -p "…"
5 new events flow through normal path
6 decision patched existing card marks "picked"

The key bit in step 3: the synthesized user message references the component by id (@dec-7). The model already saw that component in its history (it produced it), so it has the context to react. This is how old components stay live across turns — nothing about them is read-only because the conversation can always re-touch them by id.

5How HTML actually gets produced

There is no template engine and no React. The page state is exactly the fold over a mutation log. Each mutation either appends a component, patches one by id, or streams more content into one. The browser keeps a Map<id, { type, props, dom }> and applies mutations as they arrive.

The four mutation ops

// 1. Append a new component to the stack
{ op: "append", parent: "stream", component: { id: "plan-3", type: "plan", props: {…} } }

// 2. Patch a component's props (used when tool_result arrives)
{ op: "patch",  id: "plan-3", props: { items: [...] } }

// 3. Stream content into a component (Bash output, etc.)
{ op: "stream", id: "term-7", append: "line of output\n" }

// 4. Mark a component as done (terminal exit code, test pass/fail count)
{ op: "finalize", id: "term-7", meta: { exit: 0 } }

How a component is born — the producing process

  1. An agent event comes in (tool_use, tool_result, or text block).
  2. The mapper looks up the handler for that event (built-in table or render_* schema).
  3. Handler returns one or more typed mutations (append, patch, stream).
  4. Each mutation is written to .stack/sessions/<id>.events.jsonl and pushed to all browser tabs subscribed via SSE.
  5. In the browser, the component renderer for that type reads props and produces DOM. Same renderer handles initial mount and subsequent patches — the demo's refreshPlan(id) pattern, generalized.
  6. On a fresh page load, the server's /events endpoint replays the entire log before going live. The page reaches the same final state no matter when you join.
Why no framework. Components are independent of each other, each owns its own mutation logic, and the renderer-per-type pattern fits ~30 lines of vanilla JS per component. React would add bundle weight and a diffing layer for something that is already a single-direction diff stream.

6From sessions to the project page

Each session writes a per-session event log. The project page is the aggregate view across all sessions in .stack/sessions/, plus live git/CI state. It is itself a stack of components, generated fresh on each load (it's cheap — these aren't live-mutating).

Aggregation · session logs → project page
inputs session logs .stack/sessions/*.events.jsonl
inputs git state branches · uncommitted · diff vs main
inputs gh / CI PRs · workflow runs · review comments
build aggregator scans, summarizes, ranks
render project.html component stack

What the aggregator scans for in each session log:

  • Decisions resolved or open — every decision component with its picked answer.
  • Files touched — collected from diff components.
  • Test status — last tests finalize event.
  • Outcome — was the session abandoned, completed, or did it ship?
  • Title + summary — the agent's last note, or a one-line synthesis.
